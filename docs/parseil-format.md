# RouterOS parseIL — Format Reference

> Reference for the intermediate representation produced by RouterOS's `:parse`
> command — what we call **parseIL** in this project. Generated by the
> `[research: parseil]` spike (see `BACKLOG.md`); grounded on a 913-script
> corpus captured against CHR 7.22.1, then re-run on 7.20.8 and 7.23rc1 for
> drift. 912 scripts produced IL on all three versions; the one failure was the
> intentional 126 KiB oversize sample hitting the REST upload cap.
>
> Status: working reference, RouterOS 7.22.1 grammar with cross-version drift
> notes from 7.20.8 and 7.23rc1. Every snapshot in `test-data/` carries the
> version it was captured against (e.g. `foo.rsc.v7.22.1.parseil`) so drift
> stays diffable.

## 1. What parseIL is

When RouterOS evaluates a script, it first lowers the source to a textual
intermediate representation (IR) and then walks that IR. `:parse < expression >`
returns this IR as a value of type `code`. The same IR is also what the engine
serialises into `/system/script/environment` when a parsed value is bound to a
global, which is how parsed code crosses RouterOS's internal process boundary.

Concretely:

```routeros
[admin@MikroTik] > :put [:parse ":put hello"]
(evl /putmessage=hello)
```

The string `(evl /putmessage=hello)` *is* the IL — RouterOS executes that form
directly, it isn't displayed as a debugging convenience.

We capture parseIL because it gives a second, independent grounding for what
RouterOS thinks a script is, alongside the per-character `highlight` stream from
`/console/inspect`. The two are complementary: highlight tells us where every
character belongs (token classes); parseIL tells us what the script means
(structure, scope, evaluation order).

## 2. How to read out parseIL

`:parse` returns a `code`-typed value. The `code` type stringifies to the literal
placeholder `(code)` for most paths — only `:put` reveals the underlying IL:

| Readout                                     | What you get                       | Useful?           |
| ------------------------------------------- | ---------------------------------- | ----------------- |
| `:put [:parse $script]`                     | The IL string verbatim             | **Yes** — winner  |
| `:tostr [:parse $script]`                   | Literal string `(code)`            | No                |
| `:typeof [:parse $script]`                  | `code`                             | Type check only   |
| `:serialize to=json value=[:parse $script]` | JSON string `"(code)"`             | No                |
| `:global X [:parse …]; /environment print`  | `value=(code)` for the global `X`  | No                |

So the only path that surfaces the actual IL is **`:put [:parse …]`** — and to
capture that over the REST API we use `/rest/execute` with `as-string: "true"`,
which turns the synchronous `:put` output into the response body.

For non-trivial scripts the safest invocation avoids RouterOS string-escape
collisions by uploading the script to `/file/` first:

```text
POST /rest/file/add        {"name":"probe.rsc","contents": "...the script..."}
POST /rest/execute         {"script":":put [:parse [/file/get probe.rsc contents]]","as-string":"true"}
                           → {"ret": "<the IL>"}
POST /rest/file/remove     {"numbers":"probe.rsc"}
```

This is what `scripts/collect-parseil.ts` does for every `.rsc` in `test-data/`.

### Limits

- `/rest/file/add` returns **413 Payload Too Large** for files over the REST
  upload cap (observed at 126 KiB; precise threshold unmeasured). For oversize
  scripts the REST roundtrip path doesn't work; need an alternative (e.g.
  splitting the file).
- `:parse` itself does **not** appear to enforce the 32 KB cap that
  `/console/inspect` does. A 56 KB script parsed cleanly to a 56 KB IL on
  RouterOS 7.22.1, while highlight on the same content cliffs at ~28 KB.
- Parse time scales sub-quadratically with input size in the corpus: most
  scripts under 4 KB parse in ≤10 ms; the largest script in the corpus
  (56 KB `eworm/global-functions.rsc`) parsed in 1361 ms. There is no abrupt
  cliff like highlight's 28 KB step.

## 3. Grammar

The IL is text. It is **not** a JSON, S-expression, or any other standard
format, but it is well-balanced and easy to scan with a small recursive
descent parser.

### 3.1 Top-level

A parsed script is a `;`-separated sequence of zero or more **statements**:

```text
(evl …)
(evl …);(evl …)
(evl …);(evl …);(evl …)
```

Special degenerate forms:

- **Empty input → empty IL.** A 0-byte source produces a 0-byte IL.
- **Comment-only / no-op input → `/`.** A script that contains only comments
  and whitespace produces the single character `/`.

### 3.2 The `(evl …)` form — command invocation

The `evl` form is the only "instruction" in the IL. Its surface is:

```text
(evl <PATH><ARGS>)
```

with **no whitespace** between `<PATH>` and `<ARGS>`. The path/args boundary is
implicit in the grammar: the path is the maximal `/`-prefixed slug and args
start at the first `name=…` pair. To split them deterministically you need the
RouterOS command schema (which is exactly what `/console/inspect` provides).

- `<PATH>` is a fully qualified, slash-prefixed RouterOS command path.
  Examples from the corpus:
  - `/put`, `/log/info`, `/local`, `/global`, `/set`, `/if`, `/for`, `/foreach`,
    `/while`, `/do`, `/return`, `/error`, `/parse`, `/typeof`, `/tostr`,
    `/tonum`, `/toarray`, `/totime`, `/tonsec`, `/len`, `/pick`, `/find`,
    `/range`, `/nothing`, `/delay`
  - Menu commands fully qualify too:
    `/ip/firewall/address-list/add`, `/ip/route/find`, `/system/clock/get`,
    `/ipv6/firewall/filter/add`
- `<ARGS>` is `name=value` pairs joined by `;`. Names are command-specific.
  Examples:
  - `name=$x;value=42`            (after `:local x 42`)
  - `condition=(= 1 1);do=;…`     (after `:if (1=1) do={…}`)
  - `counter=$i;do=;…;in=…`       (after `:foreach i in=… do={…}`)
  - `from=1;to=3;counter=$i;do=…` (after `:for i from=1 to=3 do={…}`)

> **Top observed paths in the 7.22.1 corpus:** `/local`, `/set`, `/if`, `/put`,
> `/global`, `/return`, `/typeof`, `/tostr`, `/tonum`, `/global do=…`,
> `/foreach`, `/error`, `/find`, `/for`, `/while`, `/do`, `/log/info`,
> `/system/clock/get`, `/ip/firewall/address-list/…`. The path "shape" (how
> menu paths nest) matches `/console/inspect`'s view of RouterOS, so a path
> seen in parseIL can always be looked up there.

### 3.3 Block bodies — `do=`, `else=`, `on-error=`

When an argument value is itself a block (`do={…}`, `else={…}`,
`on-error={…}`), the IL writes the *argument key* with an empty value — `do=`
followed immediately by `;` — and then materialises the block body as a sibling
form positioned adjacently:

```routeros
:if (1=1) do={:put yes} else={:put no}
```

becomes

```text
(evl /ifcondition=(= 1 1);do=;(evl (evl /putmessage=true));else=;(evl (evl /putmessage=false)))
```

Note `do=;(evl …)` and `else=;(evl …)` — the block body lives *outside* the
arg's value lexically but is bound to the preceding `=;` argument by adjacency.

A block of more than one statement uses an `(evl <body1> <body2> …)` wrapper
where children are space-separated:

```routeros
{:put one; :put two}
```

becomes

```text
(evl (evl /putmessage=one) (evl /putmessage=two))
```

A block may also start with a bare `/` if the first statement was a comment:

```text
(evl / (evl /globalname=$PIANO) (evl /localname=$nms;value=00:00:00.200) …)
```

### 3.4 Expressions — `(<op> …)`

Operators and built-in conversions in expression position use a Lisp-style
prefix form `(<op> <arg1> <arg2> …)`:

| Operator       | Examples in the IL                        |
| -------------- | ----------------------------------------- |
| Arithmetic     | `(+ 1 2)`, `(- $a 1)`, `(* 2 3)`, `(/ 60000 …)` |
| Comparison     | `(= 1 1)`, `(< $i 3)`, `(<= 2 3)`, `(!= $a $b)` |
| Logical        | `(and (< 1 2) (<= 2 3))`, `(or … …)`      |
| String concat  | `(. "now=" $now)`                         |
| Regex match    | `(~ $dst-address 10.0.0.0)`               |
| Type-set match | `(~ (evl (evl /typeofvalue=$ms)) (num\|str))` |

Built-in conversions (`:tostr`, `:tonum`, `:totime`, `:typeof`, `:len`, etc.)
are wrapped in nested `(evl (evl /<name>value=…))` — they're commands like
anything else, just one parameter deep.

### 3.5 Quote/op values and function calls — `(> …)` and `(<%% …)`

RouterOS exposes two undocumented-but-real source-level operators that also
appear in parseIL:

| Form      | `/console/inspect` text | Runtime type/effect |
| --------- | ----------------------- | ------------------- |
| `(> …)`   | `quote`                 | Produces an `op` value without executing it immediately. |
| `(<%% …)` | `activate in environment` | Applies an op/function in an explicit argument environment. |

Both labels come directly from `request=completion` on RouterOS 7.22.1:

```text
input="("     -> completion ">"   text="quote"
input="($fn <" -> completion "<%%" text="activate in environment"
```

The quoted value has RouterOS type `op`:

```routeros
:put [:typeof (>[:put hello])]
# op
```

Executing a quoted command requires command-substitution brackets:

```routeros
[(>[:put hello])]
# hello
```

Calling a `code`/`op`-typed function normally lowers to the `<%%` form:

```routeros
:local f do={:put $1}; [$f "world"]
```

becomes (whitespace adjusted for readability):

```text
(<%% (evl
        (evl /localdo=;(evl (evl /putmessage=$1));name=$f);
        (<%% $f (> $f);world))
)
```

The inner `(<%% $f (> $f);world)` is the actual call: `$f` is the callee, and
the `(> $f)` quote marker appears before positional/named args in normal
function-call IL. Direct source-level use of `<%%` skips that inserted
callee-quote and supplies the environment explicitly:

```routeros
:global x (>[:put "$0 $1 $2 $3 $named"])
($x <%% {1;2;3;4;named="blah"})
# 1 2 3 4 blah
```

The 7.22.1 parseIL corpus contains both forms heavily (`(<%%`: 263 occurrences,
`(> `: 281 occurrences), and the source corpus includes direct uses from
forum-derived examples (`qkeys`, `$INQUIRE`, `$mkvlan`, and "What does op type
do?"). These are not IL-internal markers.

The outer `(<%% …)` wraps any IL that contains a function-call expression at
the script root, even when there is no explicit function definition. Under the
documented inspect label, that is best read as "activate the root expression in
an environment so dynamic dispatch resolves at the right scope."

### 3.6 Variable references and literals

- **Variable references:** `$name` everywhere, both scalar refs and the targets
  of `name=$x` argument bindings. There is **no syntactic distinction** between
  `:local`-bound, `:global`-bound, function-parameter, and built-in `$1`/`$2`
  references in the IL; resolution is by name, not by slot. *(This is the
  gating answer for definition/references: scope analysis from the IL alone
  requires reconstructing scopes from the surrounding `(evl /local…)` /
  `(evl /global…)` declarations.)*
- **String literals:** bare unquoted (`hello`, `MikroTik`). Strings that
  contained whitespace in source are written without quotes — whitespace
  inside an arg value is allowed and does not terminate the value.
- **Numbers:** bare unquoted (`42`, `1`, `100`).
- **Boolean canonicalisation:** `:put yes` becomes `=true`; `:put no` becomes
  `=false`. The `yes`/`no` keywords are coerced to `true`/`false` at parse
  time.
- **Time canonicalisation:** time literals are expanded to canonical
  RouterOS time form. `200ms` becomes `00:00:00.200`. This is one of the
  more useful surfaces for hover hints — the IL shows the value the engine
  will actually use.
- **Path canonicalisation:** every command path is fully qualified. `:put`
  becomes `/put`, `/ip address print` becomes `/ip/address/print`. There is
  no implicit "current menu" in the IL.
- **Argument-list dump in `find` / `where`:** `[/ip/route/find dst-address~"…"]`
  dumps **every** field of the menu it operates on as `$fieldname` markers,
  e.g. `where=$dst-address;$routing-table;$pref-src;…;$disabled;$comment;…`.
  This appears to be the IL exposing the field set the runtime reserves for
  filter-expression binding. This field dump is version-sensitive and turned out
  to be the main source of cross-version IL drift in the corpus.

## 4. Errors

`:parse` is a hard parser: at the **first** error it stops, produces no IL,
and returns only an error string with a line/column indicator:

```text
syntax error (line 6 column 3)
expected end of command (line 1 column 16)
missing closing brace (line 12 column 1)
```

The corpus capture did not intentionally feed malformed scripts through
`:parse`: 912/913 files produced IL and the one failure was the 126 KiB upload
limit before parsing. Focused negative probes produced these fatal parse-error
stems:

| Message stem             | Meaning |
| ------------------------ | ------- |
| `syntax error`           | Generic malformed expression/command syntax. |
| `expected end of command` | Extra tokens after a complete command. |
| `missing closing brace`  | Unterminated block or array literal. |
| `missing value for where` | Incomplete `where` filter expression. |

There is **no partial IL** alongside an error message. This confirms parseIL
cannot back multi-error diagnostics — for that goal, `/console/inspect
highlight` (which does mark every error token and continues) remains the source
of truth.

There is one special case that does embed errors *inside* IL: a script with a
**bad command name** like `this is not a command` produces a `(<%% …)` form
with the error annotation in-line, plus the offending tokens as positional
args:

```text
(<%% bad command name this (line 1 column 1) this;is;not;a;command)
```

That is, "bad command name" is treated as a runtime/late-binding error, while
"syntax error" is a parse-time fatal.

Error wording and `(line N column M)` coordinates are also **not** stable across
versions. The negative-probe corpus and the 7.20.8/7.22.1/7.23rc1 captures show
the same malformed source can move between `expected end of command`, `syntax
error`, and more specific `bad parameter …` strings as RouterOS tightens a menu
or reclassifies a parse failure.

## 5. Snapshot format and on-disk layout

`scripts/collect-parseil.ts` writes two files alongside each `.rsc`:

```text
test-data/.../foo.rsc
test-data/.../foo.rsc.v<routeros-version>.parseil          ← IL string, verbatim
test-data/.../foo.rsc.v<routeros-version>.parseil.meta.json ← capture metadata
```

The version suffix lets multiple RouterOS versions coexist in the corpus so IL
grammar drift across releases stays diffable. A summary file
`test-data/parseil-summary.v<version>.json` records corpus-wide stats per
capture run.

`.meta.json` shape:

```json
{
  "source": "edge-cases/single-command.rsc",
  "routerosVersion": "7.22.1",
  "chrBuildTime": "2026-03-23 14:35:15",
  "inputBytes": 18,
  "ilBytes": 23,
  "parseMs": 1,
  "ok": true,
  "capturedAt": "2026-04-25T19:33:59.530Z"
}
```

Failed captures use the same shape with `ok: false`, zero `ilBytes` when no IL
was produced, and an `error` string.

### 5.1 Cross-version drift (7.20.8 / 7.22.1 / 7.23rc1)

The full 913-script corpus was captured on RouterOS 7.20.8, 7.22.1, and
7.23rc1. Pairwise diff counts:

| Compare | Differing `.parseil` files | Shared failure |
| ------- | -------------------------- | -------------- |
| 7.20.8 ↔ 7.22.1 | 55 | `edge-cases/oversize-32k.rsc` |
| 7.22.1 ↔ 7.23rc1 | 30 | `edge-cases/oversize-32k.rsc` |
| 7.20.8 ↔ 7.23rc1 | 74 | `edge-cases/oversize-32k.rsc` |

Across all three versions, **838/912 successful captures (91.9%) are
byte-identical**. The remaining 74 files drift, but the drift is concentrated
in a few surfaces:

- **Live menu/property schema leakage.** `find where` and similar forms inline
  the current menu's field set, so added/removed properties change the IL even
  when the source is unchanged. Example: `/ip/route/find where …` differs as
  properties like `suppress-hw-offload` and `vrf` appear/disappear between
  versions.
- **Path canonicalisation.** RouterOS may fully qualify the same source command
  differently as the command tree evolves. Example: `/ping` on 7.22.1 lowers to
  `/ping`, while 7.23rc1 lowers the same source to `/tool/ping`.
- **Version-specific command/parameter validation.** Some snippets flip between
  accepted IL and in-line `bad command name …` / `bad parameter …` annotations
  as menus change. Example: `/interface bridge add add-dhcp-option82=yes …`
  lowers as a normal add on 7.22.1 but becomes `bad parameter add-dhcp-option82`
  on 7.23rc1.
- **Error-string churn.** Invalid snippets keep the same overall "no partial IL"
  rule, but message stems and column numbers move between versions.

What did **not** drift in these passes: the core IL surface. No new top-level
delimiters or forms appeared; statement sequencing still uses `;`, commands
still lower to `(evl …)`, prefix operators still use `(<op> …)`, and quoted /
activated function values still use `(> …)` and `(<%% …)`.

### 5.2 `findwhere=` field lists and required-attribute discovery

**Context:** The cross-version drift study exposed that `findwhere=` field dumps change as RouterOS adds or removes menu properties. The natural question is whether the list of fields present in a dump encodes any "required vs optional" signal for the LSP.

**What `findwhere=` actually contains.** When RouterOS compiles a `find where …` expression it emits a dump of every property of the target menu, not just the ones referenced in the filter. For example, `[/ip/address find]` produces `findwhere=$address;$actual-interface;$disabled;$dynamic;$interface;$invalid;$network;…` — that is all 15+ properties of `/ip/address`, whether readable/writable or read-only.

From the corpus (7.22.1, 56 distinct paths with `findwhere=` observed):

| Category | Contents | Present in `findwhere=`? |
| -------- | -------- | --- |
| Read-write (settable via `add`/`set`) | `chain`, `dst-address`, `interface`, `comment`, … | ✓ always |
| Read-only / computed | `.id`, `.dead`, `.about`, `active`, `running`, `dynamic`, … | ✓ always |
| Required args (confirmed empirically) | `chain` (filter), `interface` (dhcp-client), `address`+`interface` (ip/address), … | ✓ always |
| Optional args | everything else in `add` | ✓ (indistinguishable from required) |
| Positional-insertion args | `place-before` (firewall/filter/nat/mangle, bridge/port, wireguard/peers) | ✗ never |

The last row is the only structural asymmetry: `place-before` appears in `add` args from `/console/inspect` but is **never present** in the `findwhere=` dump, because it controls insertion position at creation time and is not stored as a property of the row.

**No required-arg signal in `findwhere=` or the inspect API.** Cross-referencing the full 24-path drift table against confirmed required args (sourced below) shows:

- Every confirmed required arg appears in `findwhere=` ✓ — but so do all the optional args.
- Every confirmed required arg appears in `deep-inspect.json` under `add` ✓ — but again, so do all optional args.
- `/console/inspect request=completion` returns every `add` arg with identical metadata: `preference=96`, `show=true`, `style=arg`. There is no `required`, `mandatory`, or priority-differentiated field in the completion payload.
- `deep-inspect.json` arg nodes carry only `_type: arg`, optional `desc` (human-readable type hint like `"string value"`), and optional `_completion` (enumerated values). No `required` field.

The `findwhere=` field dump therefore tells us *which properties exist for a menu in a given RouterOS version* — valuable for version-aware property enumeration — but it cannot distinguish required from optional.

**The execute-error technique: the one reliable required-arg signal.** Running `{menu} add` with no arguments causes RouterOS to produce a structured error:

```text
Script Error: missing value(s) of argument(s) <arg1> <arg2> … (<path>/add; line 1)
```

This error is exact and machine-parseable. Tested against 18 menus on 7.22.1:

| Menu | Required args (from error) |
| ---- | ------------------------- |
| `/ip/firewall/filter` | `chain` |
| `/ip/firewall/nat` | `chain` |
| `/ip/firewall/mangle` | `chain` |
| `/interface/vlan` | `interface`, `vlan-id` |
| `/interface/wireguard/peers` | `allowed-address`, `interface` |
| `/interface/bridge/port` | `bridge`, `interface` |
| `/interface/bridge/vlan` | `bridge`, `vlan-ids` |
| `/ip/hotspot/user` | `name` |
| `/ip/dhcp-server` | `interface` |
| `/ip/dhcp-client` | `interface` |
| `/system/logging/action` | `name`, `target` |
| `/ip/address` | `address`, `interface` |
| `/disk` | `type` |
| `/ip/route` | *(none — creates a blank route)* |
| `/interface/bridge` | *(none)* |
| `/interface/wireguard` | *(none)* |
| `/system/logging` | *(none)* |

Caveats: (a) `certificate` uses a non-standard error format ("At least one field specifying certificate name must be set!"), so not every menu follows the `missing value(s)` pattern; (b) `/disk` only exposes the top-level `type` requirement — conditional args like `iscsi-address` (required when `type=iscsi`) need a second pass after providing the discriminator; (c) `add` commands that don't exist (e.g. `/ip/service`) return `bad command name add` instead.

**Implications for the LSP.** The execute-error technique is an empirical, per-version probe — and that follow-up landed as [`docs/required-args.md`](required-args.md) plus `scripts/collect-required-args.ts`. A version-tagged required-arg map now exists for 7.20.8 / 7.22.1 / 7.23rc1 and is strong enough for an offline "missing required argument" diagnostic on the exact-pattern subset, without any structural change to the RouterOS inspect API.

The `findwhere=` cross-version drift (§5.1) pairs with this: when a property disappears from the drift (e.g. `suppress-hw-offload` gone from `/ip/route` on 7.22.1), confirming it was optional avoids a false-positive required-arg diagnostic on the older version's corpus.

## 6. What parseIL does and doesn't give us

### Useful for

- **Folding ranges.** Block bodies are unambiguously delimited in the IL by the
  `(evl <child> <child>…)` adjacency under `do=;`/`else=;`/`on-error=;` arg
  positions. Mapping back to source ranges still requires our own offset
  tracking, but the IL structure tells us *which* source ranges are blocks.
- **Document Symbols (functions).** A `:local f do={…}` shows up as
  `(evl /localdo=;…;name=$f)` — a clean, structural marker for "this name binds
  a callable" that the current `:local`-only document-symbols extractor misses.
- **Hover-time canonicalisation hints.** Time literals (`200ms` →
  `00:00:00.200`), boolean coercions (`yes` → `true`), path expansions
  (`:put` → `/put`) are all visible in the IL — perfect for "RouterOS will see
  this as…" hover annotations.
- **`:parse` as a cheap pre-validation.** Parse time is roughly flat under 10 ms
  for typical scripts and never exhibits highlight's 28 KB cliff. A
  `:parse`-first probe could short-circuit doomed highlight requests for known
  syntax errors at near-zero cost.
- **Debug surface ("Show parseIL").** Surfacing the IL as a read-only side
  panel or hover supplement gives users — especially LLM agents — a view of
  what RouterOS actually saw, which is the most direct debugging signal we
  could ship.

### Not useful for

- **Multi-error diagnostics.** `:parse` aborts at the first error. Highlight is
  the right source for "show every error in the script."
- **Source position metadata for valid IL.** The IL only carries `(line N
  column M)` on error annotations; valid IL nodes have no source positions, so
  mapping IL nodes back to source ranges requires structural reconstruction
  (or capturing both highlight + parseIL and aligning by token).
- **Definition / references resolution by itself.** Variable references are
  textual `$name`s in the IL with no slot information. To resolve `:local`
  vs `:global` vs function-param scoping, you still have to walk the IL and
  reconstruct scopes from the `/local`, `/global`, and `do=` boundaries.
- **Parsing without the command schema.** The IL fuses path and args without
  whitespace (`/localname=$x` is path `/local` + args `name=$x`). Splitting
  them deterministically requires knowing the command's parameter list, which
  comes from `/console/inspect` data.

## 7. LSP feature payoffs (for the BACKLOG)

The features unlocked or clarified by this spike — to be filed as concrete
backlog items once the spike is closed:

- **Folding Ranges** — straightforward off `do=;(evl …)` boundaries.
- **Document Symbols: functions, not just variables** — detect
  `(evl /localdo=…;name=$f)` / `(evl /globaldo=…;name=$F)`.
- **"Show parseIL" command + hover supplement** — verbatim IL display.
- **Hover canonicalisation hints** — display `200ms → 00:00:00.200`,
  `yes → true`, `:put → /put`.
- **`:parse` short-circuit before highlight** — skip highlight when `:parse`
  reports a fatal syntax error; surface that error directly with line/col.
- **Definition / References (with caveat)** — feasible via IL scope
  reconstruction; not free.
